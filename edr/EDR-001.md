# EDR-001: Document Numbering Strategy

**Status:** Accepted

**Date:** 2026-07-02

---

# Purpose

This document defines the official document numbering strategy used throughout WBOS.

Every business document must follow a consistent numbering system.

Document numbers are intended to be:

* Human readable
* Unique
* Predictable
* Configurable
* Auditable

---

# Philosophy

A document number is a business identifier.

It is not the database primary key.

The primary key exists for the software.

The document number exists for people.

Customers, suppliers, warehouse staff, accountants, and managers should all be able to recognize and reference documents easily.

---

# General Format

The default format is:

```text
PREFIX-YEAR-SEQUENCE
```

Example:

```text
INV-2026-000001
```

Where:

| Component | Meaning              |
| --------- | -------------------- |
| INV       | Document Type Prefix |
| 2026      | Calendar Year        |
| 000001    | Sequential Number    |

---

# Supported Document Types

Phase 1 includes:

| Type            | Prefix |
| --------------- | ------ |
| Invoice         | INV    |
| Purchase Order  | PO     |
| Credit Note     | CN     |
| Payment Receipt | PAY    |
| Goods Receipt   | GRN    |

Future document types should reuse the same numbering engine.

Examples:

* Quotations (QUO)
* Supplier Returns (SRT)
* Customer Returns (CRT)
* Warehouse Transfers (WT)
* Stock Adjustments (ADJ)

---

# Sequence Rules

Each document type maintains its own sequence.

Example:

Invoices

```text
INV-2026-000001
INV-2026-000002
INV-2026-000003
```

Purchase Orders

```text
PO-2026-000001
PO-2026-000002
```

Credit Notes

```text
CN-2026-000001
```

Sequences are completely independent.

---

# Yearly Reset

The default strategy is:

**Reset every calendar year.**

Example:

2026

```text
INV-2026-000001
...
INV-2026-004382
```

2027

```text
INV-2027-000001
```

This improves readability and simplifies accounting periods.

---

# Multi-Tenant Strategy

Document numbers are **not globally unique**.

Instead, uniqueness is scoped to:

* Organization
* Document Type
* Year
* Sequence

Internally:

```text
organization_id
document_type
year
sequence
```

This allows:

Organization A

```text
INV-2026-000015
```

Organization B

```text
INV-2026-000015
```

Both are valid because they belong to different Organizations.

The printed document should not expose tenant information.

---

# Database Design

WBOS should **not** store the formatted document number as the primary source of truth.

Instead, store its components.

Example:

| Field         | Value |
| ------------- | ----: |
| document_type |   INV |
| year          |  2026 |
| sequence      |    15 |

The display number is generated by formatting these values.

Benefits include:

* Easier formatting changes
* Better searching
* Better indexing
* Localization support
* Future flexibility

A computed `document_number` may be stored for convenience and indexed for fast lookup, but it is considered a presentation value derived from the canonical fields.

---

# Number Generation

Document numbers must be generated atomically.

Conceptually:

```text
Start Database Transaction

↓

Lock Sequence Record

↓

Increment Sequence

↓

Generate Document Number

↓

Create Business Document

↓

Commit Transaction
```

This guarantees uniqueness even under concurrent usage.

---

# Document Sequence Table

WBOS will maintain a dedicated sequence table.

Example:

| Organization | Type | Year | Current Sequence |
| ------------ | ---- | ---- | ---------------: |
| Org A        | INV  | 2026 |              382 |
| Org A        | PO   | 2026 |               74 |
| Org A        | CN   | 2026 |                9 |

Only this table is responsible for generating new document numbers.

---

# Configurable Settings

Each document type may define:

* Prefix
* Starting Number
* Sequence Length
* Reset Policy
* Separator

Default configuration:

| Setting         | Value  |
| --------------- | ------ |
| Prefix          | INV    |
| Separator       | -      |
| Digits          | 6      |
| Reset           | Yearly |
| Starting Number | 1      |

Future examples:

```text
INV/2026/000001

INV2026000001

I-2026-000001
```

Formatting changes should require configuration rather than code changes.

---

# Searching

Users should be able to search using:

Complete number:

```text
INV-2026-000123
```

or partial values:

```text
123

2026

INV
```

Searching should remain fast through indexed fields.

---

# Printing

Printed documents should display the formatted document number.

Examples:

```text
Invoice Number

INV-2026-000381
```

This identifier becomes the official business reference used by customers and suppliers.

---

# Import & Export

Imported documents must preserve their original document numbers whenever possible.

If numbering conflicts occur, the import workflow should:

* Detect duplicates
* Prevent accidental overwrites
* Allow administrator review

WBOS should never silently renumber imported business documents.

---

# Corrections

Credit Notes and Returns receive their own document numbers.

Example:

```text
Invoice

INV-2026-000125
```

Credit Note

```text
CN-2026-000011
```

Relationships between documents are maintained through references rather than shared numbering.

---

# Architectural Rules

The following rules apply throughout WBOS.

* Every business document has exactly one document number.
* Document numbers are immutable once issued.
* Numbers are generated atomically.
* Document numbers are never recycled.
* Formatting is configurable.
* Business users reference document numbers.
* Software uses primary keys internally.

---

# Future Considerations

Future enhancements may include:

* Branch-specific prefixes
* Warehouse prefixes
* Country-specific numbering
* Fiscal-year resets
* Custom formatting templates

The numbering engine should accommodate these requirements without redesign.

---

# Final Decision

WBOS will use a centralized document numbering engine shared across all business documents.

The default format is:

```text
PREFIX-YEAR-SEQUENCE
```

For example:

```text
INV-2026-000001
```

Document numbers exist for people.

Database identifiers exist for software.

Keeping these concerns separate provides a system that is easier to maintain, easier to search, and easier for users to understand.
