# ADR-007: Adopt Next.js App Router, Server Actions, and a Service-Oriented Application Layer

**Status:** Accepted

**Date:** 2026-07-02

---

# Context

WBOS is a modern web application built using Next.js.

A decision is required regarding how the application should structure communication between:

* The user interface
* Business logic
* Database access

Several architectural approaches were considered:

* Traditional REST API
* GraphQL
* Next.js Server Actions
* Hybrid architectures
* Separate backend service (Express, Fastify, NestJS)

The chosen solution must:

* Keep business logic centralized
* Minimize duplication
* Support future expansion
* Remain easy to maintain
* Integrate naturally with Next.js

---

# Decision

WBOS will use:

* **Next.js App Router**
* **React Server Components by default**
* **Server Actions for mutations**
* **Route Handlers when HTTP endpoints are required**
* **Application Services for all business logic**
* **Repositories for database access**

Server Actions are the transport layer.

They are **not** the business layer.

Business logic always lives inside Application Services.

---

# Architecture Overview

Every request should follow the same path.

```text
Browser
    │
    ▼
React Component
    │
    ▼
Server Action / Route Handler
    │
    ▼
Application Service
    │
    ▼
Repository
    │
    ▼
Prisma
    │
    ▼
PostgreSQL
```

Every layer has one responsibility.

Business rules never bypass these layers.

---

# Philosophy

Server Actions are simply another way to invoke business logic.

They should remain extremely thin.

Their responsibilities are limited to:

* Receiving input
* Authentication
* Authorization
* Validation
* Calling the appropriate Application Service
* Returning the result

They should never contain business workflows.

---

# Application Services

Application Services represent business use cases.

Examples include:

* ReceiveShipmentService
* CreateInvoiceService
* RecordPaymentService
* ArchiveProductService
* CreatePurchaseOrderService

Application Services coordinate business workflows.

They are the heart of WBOS.

---

# Why a Service Layer?

Business logic should exist in exactly one place.

Without a Service Layer:

* Logic becomes duplicated.
* Validation becomes inconsistent.
* APIs diverge.
* Mobile support becomes harder.
* Background jobs duplicate code.

Application Services solve these problems.

---

# Repositories

Repositories own database access.

Responsibilities include:

* Reading data
* Writing data
* Query optimization
* Mapping database models

Repositories should not contain business decisions.

They answer:

**"How do we retrieve this data?"**

Application Services answer:

**"What should happen?"**

---

# Presentation Layer

The Presentation Layer includes:

* Pages
* Layouts
* Components
* Forms
* Tables
* Dialogs

Responsibilities:

* Display data
* Collect user input
* Trigger actions

The Presentation Layer should never implement business workflows.

---

# Server Actions

Server Actions are preferred for:

* Creating records
* Updating records
* Deleting records
* Executing workflows
* Form submissions

They integrate naturally with React and reduce unnecessary API boilerplate.

---

# Route Handlers

Route Handlers remain important.

They should be used for:

* Public APIs
* Webhooks
* File downloads
* File uploads
* Third-party integrations
* Streaming responses
* Mobile clients (future)

Server Actions should not replace every HTTP endpoint.

---

# React Server Components

Server Components are the default.

Benefits include:

* Reduced client-side JavaScript
* Faster page loads
* Better SEO
* Direct database access through services
* Simpler architecture

Client Components should only be introduced when interactivity is required.

---

# Client Components

Client Components are appropriate for:

* Forms
* Interactive tables
* Drag-and-drop
* Charts
* Real-time UI
* Browser APIs

Client Components should remain focused on interaction rather than business logic.

---

# Validation

Validation occurs in multiple layers.

Client

↓

User Experience

Server Action

↓

Input Validation

Application Service

↓

Business Validation

Database

↓

Integrity Constraints

Every layer has a different purpose.

Validation should never rely solely on the client.

---

# Transactions

Complex workflows should execute inside database transactions.

Example:

Receive Shipment

↓

Validate Purchase Order

↓

Create Inventory Lots

↓

Create Inventory Transactions

↓

Update Purchase Order

↓

Write Activity Log

↓

Commit

Application Services own transaction boundaries.

---

# Error Handling

Application Services throw meaningful business exceptions.

Examples:

* InventoryUnavailableError
* CustomerCreditExceededError
* InvalidInvoiceStateError

Server Actions translate these exceptions into appropriate responses for the UI.

---

# Dependency Direction

Dependencies always point downward.

```text
Presentation

↓

Transport

↓

Application

↓

Repository

↓

Database
```

Lower layers never depend on upper layers.

Business logic should never depend on React.

---

# Why Not REST Everywhere?

REST APIs were considered.

Rejected as the primary communication mechanism because:

* Additional boilerplate
* Duplicate validation
* Extra serialization
* Slower development
* Less natural integration with Next.js

REST remains valuable for external integrations.

It is not required for internal application communication.

---

# Why Not GraphQL?

GraphQL was considered.

Rejected because:

* Additional complexity
* Schema maintenance
* Limited business value for the current scope

The business does not currently require GraphQL capabilities.

---

# Why Not Express or NestJS?

A dedicated backend framework was considered.

Rejected because:

* Additional infrastructure
* Duplicate routing
* More deployment complexity
* Less integration with React

Next.js already provides everything required.

Business logic remains framework-independent inside Application Services.

If a separate backend becomes necessary in the future, Application Services can be reused.

---

# Future Expansion

Future interfaces may include:

* Mobile application
* Public REST API
* GraphQL API
* CLI tools
* Background workers
* Scheduled jobs

Each interface should call the same Application Services.

Business logic should never be duplicated.

---

# Architectural Rules

The following rules are mandatory.

* Business logic belongs in Application Services.
* Database access belongs in Repositories.
* Server Actions remain thin.
* Components never access Prisma.
* Components never implement business workflows.
* Services remain independent of React.
* Route Handlers reuse Application Services.
* Validation occurs on both client and server.

---

# Consequences

This architecture provides:

* Clear separation of responsibilities
* Excellent testability
* Reduced duplication
* Better maintainability
* Easier future expansion
* Consistent business logic

The application remains simple while supporting future interfaces without architectural changes.

---

# Review Criteria

This decision should be reconsidered only if:

* Next.js fundamentally changes its architecture.
* Server Actions become unsuitable for business applications.
* The project requires a completely independent backend for operational reasons.

Until then, this architecture remains the preferred approach.

---

# Final Decision

WBOS will use the Next.js App Router with React Server Components, Server Actions, Route Handlers, Application Services, and Repositories.

Server Actions are a transport mechanism—not the business layer.

Application Services are the true heart of the application.

This separation ensures that business logic remains reusable, testable, framework-independent, and ready for future expansion.

The UI should describe **what the user wants to do**.

Application Services decide **how the business performs that work**.
