# ADR-002: Adopt a Multi-Tenant Data Model from Day One

**Status:** Accepted

**Date:** 2026-07-02

---

# Context

WBOS is initially being developed for a single wholesale import business.

The first production deployment will serve only our own company.

However, the long-term vision of WBOS includes the possibility of becoming a SaaS platform that can support multiple businesses.

A decision is required regarding whether the database should be designed as:

* Single-tenant
* Multi-tenant

---

# Decision

WBOS will adopt a **multi-tenant data model** from the beginning.

Every business entity will belong to exactly one **Organization**.

Examples include:

* Products
* Categories
* Customers
* Suppliers
* Warehouses
* Purchase Orders
* Shipments
* Inventory Lots
* Inventory Transactions
* Invoices
* Payments
* Business Settings

The first production deployment will contain only one Organization.

The user interface will behave exactly like a single-company application.

---

# Why This Decision Was Made

This decision is **not** driven by SaaS requirements.

It is driven by long-term maintainability.

Adding an `organizationId` to business entities during the initial design is inexpensive.

Adding tenant isolation after years of development would require:

* Database migrations
* Code refactoring
* API redesign
* Permission changes
* Extensive testing
* Risk to historical data

The cost of designing for multi-tenancy today is minimal compared to the cost of introducing it later.

---

# Scope

Multi-tenancy applies only to the data model and business logic.

It does **not** imply that Phase 1 will include:

* Organization switching
* Multi-company dashboards
* Organization invitations
* SaaS billing
* Subscription management
* Tenant administration
* Cross-organization management

These features are intentionally deferred.

---

# Organization Ownership

Every business record belongs to one Organization.

Example:

```text
Organization
│
├── Products
├── Categories
├── Customers
├── Suppliers
├── Warehouses
├── Purchase Orders
├── Shipments
├── Inventory Lots
├── Inventory Transactions
├── Invoices
├── Payments
└── Business Settings
```

Cross-organization relationships are not permitted.

---

# Data Isolation

Business data must always remain isolated.

Every database query must be scoped to the current Organization.

Access to another organization's data must never occur accidentally.

Organization boundaries are considered a fundamental security boundary.

---

# Authentication & Authorization

Users belong to one or more Organizations.

Permissions are evaluated within the context of the active Organization.

Future versions may support users belonging to multiple Organizations, but Phase 1 assumes a single active Organization per user.

---

# Query Rules

All business queries must include Organization filtering.

Example:

Good:

```sql
SELECT *
FROM products
WHERE organization_id = ?
```

Bad:

```sql
SELECT *
FROM products
```

Repositories are responsible for enforcing this rule consistently.

---

# Benefits

## Future SaaS Readiness

The data model is already prepared if WBOS evolves into a commercial SaaS platform.

No disruptive migration should be required.

---

## Cleaner Ownership

Every business record has a clear owner.

This simplifies:

* Permissions
* Reporting
* Backups
* Data exports
* Auditing

---

## Improved Security

Organization boundaries become explicit.

This reduces the risk of accidental data leakage.

---

## Better Testing

Tests can verify that data remains isolated between Organizations.

Tenant isolation becomes part of the automated test suite.

---

# Trade-Offs

The chosen approach introduces a small amount of additional complexity.

Examples include:

* Additional foreign keys
* Repository filtering
* Organization context
* Slightly more verbose queries

These costs are considered acceptable.

---

# Alternatives Considered

## Single-Tenant Database

Rejected.

Advantages:

* Slightly simpler schema
* Slightly simpler queries

Disadvantages:

* Difficult migration later
* Significant refactoring
* Higher long-term risk

The short-term simplicity does not justify the future migration cost.

---

## Separate Database Per Organization

Rejected.

Although this provides strong isolation, it introduces unnecessary operational complexity.

Managing multiple databases would complicate:

* Backups
* Migrations
* Monitoring
* Reporting
* Deployment

The expected scale of WBOS does not justify this approach.

---

# Architectural Rules

The following rules apply throughout the application.

* Every business entity belongs to exactly one Organization.
* Organization ownership cannot change after creation.
* Organization IDs are never editable.
* Cross-organization references are prohibited.
* Business services must always operate within an Organization context.
* Repository methods should require Organization context wherever applicable.

---

# Future Considerations

Future versions of WBOS may introduce:

* Multiple Organizations per user
* Organization switching
* SaaS subscriptions
* Organization invitations
* Tenant administration
* White-label deployments

These capabilities should build upon the existing Organization model rather than requiring redesign.

---

# Consequences

This decision provides a strong architectural foundation with minimal impact on the initial implementation.

The application behaves like a single-company system today while remaining prepared for future expansion.

The additional complexity is intentionally kept small and localized.

---

# Review Criteria

This decision should be revisited only if:

* WBOS is permanently limited to a single organization with no possibility of expansion.
* A fundamentally different deployment model is adopted.

At present, neither condition applies.

---

# Final Decision

WBOS will use a multi-tenant data model from the beginning.

This is **not** because we are building a SaaS product today.

It is because the cost of preparing the data model now is negligible compared to the cost of redesigning it later.

The goal is not premature optimization.

The goal is avoiding unnecessary future migration while keeping the current implementation simple.
