# ADR-006: Adopt Better Auth for Authentication and Identity Management

**Status:** Accepted

**Date:** 2026-07-02

---

# Context

WBOS is a business-critical application that manages sensitive operational and financial data.

Every action performed within the system must be attributable to an authenticated user.

The authentication system must provide:

* Secure authentication
* Session management
* Organization awareness
* Future role-based permissions
* Self-hosting compatibility
* Excellent developer experience
* Long-term maintainability

Authentication should integrate naturally with the overall architecture without introducing unnecessary complexity or vendor lock-in.

---

# Decision

WBOS will use **Better Auth** as the authentication and identity management solution.

Better Auth will manage:

* User authentication
* Sessions
* Identity
* Password management
* Organization membership
* Future role support

Business permissions and workflows remain the responsibility of WBOS.

---

# Authentication Philosophy

Authentication answers:

> **Who is this user?**

Authorization answers:

> **What is this user allowed to do?**

These are separate concerns.

Authentication belongs to infrastructure.

Authorization belongs to the business domain.

---

# Identity Model

The primary identity hierarchy is:

```text
Organization
        │
        ▼
      User
        │
        ▼
      Session
```

Every authenticated request operates within the context of one Organization.

---

# Why Better Auth?

Better Auth aligns closely with the goals of WBOS.

It provides:

* Modern architecture
* TypeScript-first development
* Self-hosting support
* Excellent integration with Next.js
* Flexible session management
* Future extensibility

Most importantly, it allows WBOS to own its authentication rather than relying on a third-party identity provider.

---

# Self-Hosting

WBOS is designed primarily for self-hosted deployments.

Better Auth allows authentication to remain entirely under our control.

Benefits include:

* No external dependency
* No recurring authentication costs
* Full ownership of user data
* Consistent deployment experience

Development and production environments behave the same way.

---

# Session Management

Authentication should be session-based.

Sessions should:

* Be secure
* Be revocable
* Expire automatically
* Support multiple devices

Session handling should be transparent to users while remaining secure.

---

# User Accounts

A User represents a real person.

Users perform business actions.

Users do not own business data.

Instead, they act on behalf of an Organization.

Examples:

* Creating Products
* Receiving Shipments
* Issuing Invoices
* Recording Payments

Every important action should be traceable to the responsible User.

---

# Organization Context

Although WBOS initially supports one Organization, every authenticated request should include Organization context.

Benefits include:

* Data isolation
* Future SaaS support
* Cleaner permission checks
* Better auditability

The current Organization should be resolved automatically after authentication.

---

# Authorization Strategy

Authentication identifies users.

Authorization determines permissions.

WBOS will implement authorization independently from Better Auth.

This keeps business rules independent of the authentication provider.

---

# Future Role-Based Access Control (RBAC)

Future roles may include:

* Owner
* Administrator
* Manager
* Sales
* Warehouse
* Finance
* Viewer

Permissions should be evaluated on the server.

The client should never determine whether an action is allowed.

---

# Activity Logging

Every authenticated action should record:

* User
* Organization
* Timestamp
* Action
* Target entity

Authentication supports accountability.

Activity Logs preserve business history.

---

# Password Security

Passwords should never be stored directly.

Authentication should follow industry best practices.

Requirements include:

* Secure hashing
* Rate limiting
* Session invalidation
* Password reset
* Email verification (future)

Security should rely on proven libraries rather than custom implementations.

---

# Future Authentication Options

The architecture should allow future support for:

* Passkeys
* Two-Factor Authentication (2FA)
* Single Sign-On (SSO)
* OAuth providers
* Magic links

These features should extend the existing identity model rather than replacing it.

---

# Why Not NextAuth (Auth.js)?

Auth.js was considered.

Rejected because:

* Better Auth provides a cleaner developer experience.
* Better TypeScript integration.
* More modern architecture.
* Better alignment with the project's long-term goals.

This decision reflects current project requirements and may be revisited if the ecosystem changes significantly.

---

# Why Not Clerk?

Clerk provides an excellent authentication platform.

Rejected because:

* Vendor dependency
* Cloud-first architecture
* Recurring cost
* Reduced control over identity infrastructure

WBOS prioritizes self-hosting and full ownership of authentication.

---

# Why Not Firebase Authentication?

Rejected because:

* Vendor lock-in
* Cloud dependency
* Limited alignment with self-hosting goals

WBOS should remain infrastructure-independent whenever practical.

---

# Why Not Build Authentication Ourselves?

Custom authentication was rejected.

Authentication is security-critical.

Maintaining a custom authentication system would:

* Increase maintenance burden
* Increase security risk
* Distract from core business functionality

Established authentication libraries should be preferred.

---

# Architectural Rules

The following rules apply throughout WBOS.

* Authentication is mandatory.
* Every request executes as an authenticated User.
* Every User belongs to an Organization.
* Authorization is enforced on the server.
* Business logic never trusts client-side permissions.
* Sessions must be revocable.
* Every significant action is attributable to a User.

---

# Consequences

This decision provides:

* Secure authentication
* Self-hosted identity
* Future RBAC support
* Strong auditability
* Organization awareness
* Excellent developer experience

Authentication remains isolated from business logic while supporting future growth.

---

# Future Considerations

Future enhancements may include:

* Passkeys
* 2FA
* Device management
* Session management UI
* Organization invitations
* Multiple Organizations per User
* Enterprise SSO

The chosen architecture supports these capabilities without requiring fundamental redesign.

---

# Review Criteria

This decision should be reconsidered only if:

* Better Auth no longer meets the project's security or maintenance requirements.
* A future authentication standard provides significant advantages.
* Business requirements fundamentally change.

At present, Better Auth provides the best balance of flexibility, security, developer experience, and self-hosting support.

---

# Final Decision

WBOS will adopt Better Auth for authentication and identity management.

Authentication will remain an infrastructure concern.

Authorization and business permissions will remain part of the WBOS domain model.

This separation keeps the architecture modular, secure, and adaptable while allowing the business to evolve independently of the authentication provider.

Identity establishes **who** the user is.

WBOS determines **what** that user is allowed to do.
