# ADR-008: Adopt a Domain-Driven Architecture with Service and Repository Layers

**Status:** Accepted

**Date:** 2026-07-02

---

# Context

WBOS is expected to become a long-term business platform consisting of multiple business domains including:

* Inventory
* Purchasing
* Sales
* Customers
* Suppliers
* Payments
* Warehouses
* Reporting
* Administration

As the application grows, maintaining a clear and predictable codebase becomes increasingly important.

A decision is required regarding how the project should be organized internally.

---

# Decision

WBOS will adopt a **Domain-Driven Architecture**.

Business domains are the primary organizational unit of the application.

Each domain owns:

* Business logic
* Application services
* Repositories
* Validation
* Components
* Types
* Tests

Shared infrastructure will exist outside the domains.

---

# Philosophy

The codebase should reflect the business.

Developers should think in terms of:

* Inventory
* Customers
* Purchasing
* Sales

—not—

* Components
* Hooks
* Helpers
* Utils

The folder structure should teach new developers how the business works.

---

# High-Level Structure

```text
src/
│
├── app/
│
├── domains/
│   │
│   ├── inventory/
│   ├── purchasing/
│   ├── products/
│   ├── customers/
│   ├── suppliers/
│   ├── invoices/
│   ├── payments/
│   ├── warehouses/
│   ├── reporting/
│   └── settings/
│
├── shared/
│
├── infrastructure/
│
└── lib/
```

The application is organized around business capabilities rather than technical categories.

---

# Domain Ownership

Each domain owns everything required to solve its business problems.

Example:

```text
domains/

inventory/

├── components/

├── services/

├── repositories/

├── validation/

├── types/

├── hooks/

├── utils/

├── tests/

└── index.ts
```

The Inventory domain should not depend on implementation details from unrelated domains.

---

# Application Services

Application Services coordinate business workflows.

Examples:

* ReceiveShipmentService
* CreateInvoiceService
* RecordPaymentService
* ArchiveProductService

Responsibilities include:

* Workflow orchestration
* Validation
* Transactions
* Business rules
* Coordination between repositories

Application Services answer:

> **"What should happen?"**

---

# Repositories

Repositories isolate persistence.

Responsibilities include:

* Reading data
* Writing data
* Query optimization
* Mapping database models

Repositories answer:

> **"How do we retrieve or persist data?"**

Repositories never contain business workflows.

---

# Components

Components exist only to present information.

Responsibilities include:

* Rendering UI
* Collecting input
* Displaying validation
* Triggering actions

Components must never:

* Query Prisma
* Execute business workflows
* Implement business rules

---

# Validation

Validation belongs inside each domain.

Examples:

```text
inventory/

validation/

receive-shipment.ts

adjust-stock.ts
```

Validation should remain close to the business workflow it protects.

---

# Types

Domain-specific types belong to their owning domain.

Examples:

```text
inventory/

types/

inventory.ts

inventory-lot.ts

inventory-transaction.ts
```

Shared types belong inside:

```text
shared/types/
```

---

# Tests

Each domain owns its own tests.

Examples:

```text
inventory/

tests/

receive-shipment.test.ts

fifo.test.ts

adjustment.test.ts
```

Tests should remain close to the code they verify.

---

# Shared Layer

The shared layer contains functionality used across multiple domains.

Examples include:

* UI components
* Common utilities
* Shared types
* Constants
* Error classes

Shared code should remain business-agnostic whenever possible.

---

# Infrastructure Layer

Infrastructure supports the application but does not define business behavior.

Examples:

* Prisma
* Better Auth
* File Storage
* Logging
* Email
* Search
* External APIs

Infrastructure should be replaceable with minimal impact on business logic.

---

# Dependency Rules

Dependencies always point inward.

```text
UI

↓

Application Services

↓

Repositories

↓

Infrastructure
```

Business logic should never depend on:

* React
* Prisma
* Authentication provider
* File storage implementation

Business logic should remain portable.

---

# Cross-Domain Communication

Domains should communicate through public services.

Example:

```text
Invoice Service

↓

Inventory Service

↓

Inventory Repository
```

Domains should not reach directly into another domain's private implementation.

---

# Why Domain-Driven Organization?

A traditional structure:

```text
components/

hooks/

pages/

utils/

services/
```

works well for small applications.

As the application grows, related business logic becomes scattered.

Finding everything related to "Inventory" requires searching the entire project.

A domain-first structure keeps related code together.

---

# Why Not Feature-Based?

Feature-based organization was considered.

Rejected because features evolve over time.

Business domains remain stable.

Examples:

Inventory will always exist.

Purchasing will always exist.

Customers will always exist.

The architecture should mirror long-term business concepts rather than temporary features.

---

# Why Not Layer-Based?

A layer-based structure such as:

```text
components/

repositories/

services/

controllers/
```

was rejected.

Although technically organized, it separates code that belongs together.

Business workflows become fragmented across the codebase.

Developers spend more time navigating folders than solving business problems.

---

# Benefits

This architecture provides:

* Clear ownership
* High cohesion
* Low coupling
* Easier onboarding
* Better maintainability
* Better scalability
* Simpler testing

Developers can understand an entire business capability by exploring one domain.

---

# Trade-Offs

This architecture introduces:

* Slightly more folders
* More explicit boundaries
* More deliberate organization

These trade-offs are considered worthwhile because they significantly improve long-term maintainability.

---

# Future Growth

As WBOS evolves, domains may eventually become independently deployable.

Examples include:

* Reporting
* Search
* AI
* Notifications

Because domains are already well-defined, future extraction becomes significantly easier.

The application remains a Modular Monolith until business requirements justify further separation.

---

# Architectural Rules

The following rules apply throughout WBOS.

* Business logic belongs in Application Services.
* Persistence belongs in Repositories.
* UI belongs in Components.
* Validation belongs to the owning domain.
* Tests belong to the owning domain.
* Shared code should remain generic.
* Infrastructure should remain replaceable.
* Domains should expose public interfaces rather than internal implementation details.

---

# Consequences

The project structure becomes an expression of the business itself.

Developers spend less time searching for code and more time understanding workflows.

The architecture supports both rapid development today and long-term evolution tomorrow.

---

# Review Criteria

This decision should only be reconsidered if:

* The application fundamentally changes architecture.
* Business domains cease to be the primary organizing principle.
* A demonstrably better organizational strategy emerges.

At present, a domain-driven architecture provides the clearest and most maintainable structure for WBOS.

---

# Final Decision

WBOS will organize its codebase around business domains.

Every domain owns its own business logic, services, repositories, validation, types, and tests.

Shared infrastructure remains outside the domains.

This organization reflects how the business operates rather than how the framework is implemented.

The structure of the codebase should teach developers how the business works.

That is the ultimate purpose of the architecture.
