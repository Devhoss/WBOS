# ADR-001: Adopt a Modular Monolith Architecture

**Status:** Accepted

**Date:** 2026-07-02

---

# Context

WBOS is a long-term software platform being developed to replace our Excel-based business workflow.

Although the long-term vision includes support for multiple organizations, AI features, public APIs, and third-party integrations, the initial product will be developed and maintained by a very small team.

The architecture must therefore balance:

* Simplicity
* Maintainability
* Scalability
* Developer productivity
* Long-term flexibility

A decision is required regarding the overall application architecture.

---

# Decision

WBOS will be implemented as a **Modular Monolith**.

The application will be deployed as a single application while internally being divided into independent business domains.

Each domain owns:

* Business logic
* Validation
* Services
* Components
* Types
* Tests

Examples include:

* Inventory
* Purchasing
* Customers
* Suppliers
* Payments
* Sales
* Reporting

The application will have clear internal boundaries without requiring separate deployments.

---

# Why This Decision Was Made

A Modular Monolith provides the best balance between simplicity and future growth.

It allows:

* Fast development
* Easy debugging
* Simple deployment
* Single database
* Single repository
* Shared transactions
* Reduced operational overhead

At the same time, clear domain boundaries make future extraction into independent services possible if business requirements justify it.

---

# Benefits

## Faster Development

A single application is significantly easier to develop than a distributed system.

No service discovery.

No network communication.

No API versioning between internal services.

---

## Simpler Deployment

Production deployment consists of:

* Next.js
* PostgreSQL
* Docker Compose

This aligns perfectly with the planned homelab deployment.

---

## Easier Debugging

Business workflows often span multiple domains.

Example:

Receive Shipment

↓

Inventory

↓

Purchase Order

↓

Inventory Lots

↓

Activity Log

Within a Modular Monolith these workflows execute inside a single process and can share a database transaction.

---

## Strong Data Consistency

Critical workflows require ACID database transactions.

Examples include:

* Receiving inventory
* Creating invoices
* Recording payments

Keeping these operations inside one application greatly simplifies consistency.

---

## Lower Operational Complexity

The application does not require:

* API gateways
* Service mesh
* Distributed tracing
* Message brokers
* Container orchestration
* Cross-service authentication

This allows the team to focus on business features rather than infrastructure.

---

# Trade-Offs

A Modular Monolith is not infinitely scalable.

As the application grows:

* Build times increase.
* Deployment size increases.
* Team coordination becomes more important.

However, these trade-offs are acceptable given the expected size of the project.

---

# Alternatives Considered

## Microservices

Rejected.

Reasons:

* Excessive operational complexity.
* Small development team.
* Shared database transactions become difficult.
* Infrastructure overhead outweighs benefits.

Microservices solve organizational scaling problems more than technical ones.

WBOS does not currently have those problems.

---

## Service-Oriented Architecture (SOA)

Rejected.

Would introduce unnecessary communication layers while providing little business value during early development.

---

## Separate Backend + Frontend

Rejected for Phase 1.

A dedicated backend (Express, Fastify, NestJS, etc.) adds additional maintenance without solving an immediate problem.

Next.js Server Actions and Route Handlers are sufficient for current requirements.

Business logic will remain inside a service layer so that future extraction remains straightforward if required.

---

# Architectural Rules

To preserve modularity, every domain must:

* Own its own business rules.
* Avoid direct dependencies on unrelated domains.
* Communicate through services rather than implementation details.
* Avoid leaking infrastructure concerns.

Shared utilities belong in common infrastructure rather than inside business domains.

---

# Future Evolution

If future requirements justify decomposition, domains should be extractable into independent services with minimal changes.

Examples of potential future services include:

* AI Services
* Search
* Notifications
* Public API
* Reporting

No extraction should occur until there is a demonstrated operational need.

---

# Consequences

This decision results in:

* One repository
* One deployable application
* One PostgreSQL database
* One authentication system
* Clear internal modules
* Strong transactional consistency

The architecture remains simple while leaving room for future growth.

---

# Review Criteria

This decision should only be reconsidered if:

* The development team grows significantly.
* Independent deployment of domains becomes necessary.
* Performance bottlenecks cannot be solved within the monolith.
* Business requirements require independent scaling of specific domains.

Until then, the Modular Monolith remains the preferred architecture.

---

# Final Decision

WBOS will remain a Modular Monolith unless a measurable business or technical requirement justifies a different architecture.

Complexity should be introduced only when it solves a real problem.

This decision aligns with the overall engineering philosophy of WBOS:

> Prefer simple, maintainable solutions over unnecessary architectural complexity.
