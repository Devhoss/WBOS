# ADR-005: Adopt PostgreSQL and Prisma as the Data Platform

**Status:** Accepted

**Date:** 2026-07-02

---

# Context

WBOS is a long-term business application responsible for managing inventory, purchasing, invoicing, payments, reporting, and business history.

The database platform must provide:

* Strong data integrity
* Transactional consistency
* Excellent relational modeling
* Long-term reliability
* Predictable performance
* Mature tooling
* Simple deployment
* Future scalability

The ORM must complement these goals without hiding important database concepts.

---

# Decision

WBOS will use:

* **PostgreSQL** as the primary database.
* **Prisma ORM** as the database access layer.

Prisma will be the only supported ORM.

PostgreSQL will be the single source of truth for persistent business data.

---

# Why PostgreSQL?

PostgreSQL is one of the most mature relational database systems available.

It provides everything WBOS requires while remaining open source, well documented, and highly reliable.

PostgreSQL aligns with the long-term goals of the project.

---

# ACID Transactions

Many WBOS workflows modify multiple business entities simultaneously.

Examples include:

Receive Shipment

↓

Create Inventory Lots

↓

Create Inventory Transactions

↓

Update Purchase Order

↓

Write Activity Log

These operations must either succeed together or fail together.

PostgreSQL provides reliable ACID transactions that make this possible.

---

# Strong Relational Modeling

WBOS is built around relationships.

Examples:

Organization

↓

Customers

↓

Invoices

↓

Payments

Or:

Products

↓

Inventory Lots

↓

Inventory Transactions

↓

Reports

A relational database models these relationships naturally.

---

# Data Integrity

Business software depends on trustworthy data.

PostgreSQL supports:

* Foreign Keys
* Unique Constraints
* Check Constraints
* Transactions
* Indexes

The database should actively protect business integrity.

Application code alone is not sufficient.

---

# Mature Feature Set

PostgreSQL provides numerous features that support future growth.

Examples include:

* Full Text Search
* JSON Columns
* Materialized Views
* Recursive Queries
* Window Functions
* Generated Columns
* Partial Indexes

WBOS will adopt these features only when they provide measurable value.

---

# Docker Support

PostgreSQL runs exceptionally well in Docker.

This aligns with the deployment strategy:

Development

↓

Docker Desktop

↓

Production

↓

Docker Compose

↓

Homelab

The same database engine is used everywhere.

---

# Why Prisma?

Prisma provides a modern, type-safe developer experience while allowing PostgreSQL to remain the source of truth.

It increases developer productivity without hiding relational concepts.

---

# Type Safety

Prisma generates TypeScript types directly from the schema.

Benefits include:

* Autocomplete
* Compile-time validation
* Safer refactoring
* Reduced runtime errors

Strong typing aligns with WBOS engineering standards.

---

# Schema Management

Prisma provides declarative schema management.

The schema becomes version-controlled alongside the application.

Every schema change is reviewed, documented, and migrated consistently.

---

# Migrations

Database changes should never be performed manually.

Every structural change should be introduced through version-controlled migrations.

Benefits include:

* Repeatable deployments
* Reliable upgrades
* Team consistency
* Easier rollback planning

---

# Developer Experience

Prisma offers an excellent development experience.

Features include:

* Schema validation
* Typed queries
* Database introspection
* Migration tooling
* Prisma Studio

These tools improve development speed while maintaining quality.

---

# Repository Pattern

Prisma should never be used directly throughout the application.

Instead:

```text
UI

↓

Server Actions

↓

Application Services

↓

Repositories

↓

Prisma

↓

PostgreSQL
```

Repositories isolate Prisma from business logic.

This allows the ORM to change in the future with minimal impact.

---

# Performance Philosophy

Prisma is selected for maintainability, not because it is the fastest ORM available.

Performance should be measured.

If necessary:

* Optimize queries.
* Introduce indexes.
* Use raw SQL selectively.

Raw SQL should be reserved for cases where measurable performance benefits exist.

---

# Database Ownership

PostgreSQL owns persistent business data.

Prisma manages access to that data.

Business logic belongs inside services.

The database should never become a dumping ground for application logic.

---

# Why Not MySQL?

MySQL was considered.

Rejected because:

* PostgreSQL offers a richer feature set.
* Better support for advanced SQL.
* Stronger ecosystem for analytical queries.
* More flexibility for future reporting.

Both are capable databases, but PostgreSQL better aligns with WBOS's long-term goals.

---

# Why Not SQLite?

SQLite is excellent for small applications.

Rejected because:

* Limited concurrency.
* Less suitable for multi-user business software.
* More difficult long-term scaling.

SQLite is appropriate for prototypes, not the production vision of WBOS.

---

# Why Not MongoDB?

MongoDB was rejected because WBOS is fundamentally relational.

Examples:

* Customers
* Invoices
* Payments
* Purchase Orders
* Inventory Lots
* Inventory Transactions

These relationships are central to the business model.

A relational database represents them naturally.

---

# Why Not Supabase?

Supabase was considered.

Rejected for the initial architecture because:

* WBOS is self-hosted.
* PostgreSQL can be managed directly.
* We do not currently require Supabase-specific platform features.

Future deployments could use Supabase if business requirements change, as it is built on PostgreSQL.

---

# Why Not Drizzle ORM?

Drizzle is an excellent ORM.

Prisma was selected because:

* More mature migration tooling.
* Excellent schema-first workflow.
* Outstanding TypeScript integration.
* Larger ecosystem.
* Familiar developer experience.

This decision is based on current project needs rather than deficiencies in Drizzle.

---

# Why Not TypeORM?

TypeORM was rejected because:

* More complex configuration.
* Less predictable type safety.
* Decorator-heavy API.
* Higher maintenance overhead.

Prisma provides a simpler and more explicit development experience.

---

# Architectural Rules

The following rules apply:

* PostgreSQL is the only supported production database.
* Prisma is the only ORM.
* Database access occurs only through repositories.
* Business logic never depends directly on Prisma.
* Migrations are version-controlled.
* Schema changes require review.
* Raw SQL should be exceptional, not standard.

---

# Future Considerations

Future enhancements may include:

* Read replicas
* Materialized views
* Full Text Search
* Partitioning
* Background reporting
* Database monitoring

These enhancements should build upon PostgreSQL rather than replacing it.

---

# Consequences

This decision provides:

* Strong relational modeling
* Reliable transactions
* Excellent developer experience
* Predictable deployments
* High maintainability
* Long-term flexibility

The combination of PostgreSQL and Prisma aligns with both the business requirements and the engineering philosophy of WBOS.

---

# Review Criteria

This decision should only be reconsidered if:

* PostgreSQL no longer satisfies business requirements.
* Prisma becomes a significant maintenance burden.
* A measurable technical limitation cannot be resolved within the current architecture.

At present, neither condition exists.

---

# Final Decision

WBOS will use PostgreSQL and Prisma as its data platform.

PostgreSQL provides the reliability, consistency, and relational capabilities required for business-critical software.

Prisma provides a modern, type-safe development experience without compromising the underlying database design.

The combination supports our guiding principles:

* Correctness over convenience.
* Maintainability over cleverness.
* Long-term reliability over short-term speed.

The database is the foundation of WBOS.

Choosing a mature, proven, and well-understood platform is one of the most important engineering decisions we can make.
